# Support-Donor Notes — freezebreeze_live_lab_ice_blue_v1

How this pack should *support* other packs and tasks as a secondary donor, beyond serving as the primary live-commerce anchor.

## What this pack is the canonical donor for

- **Live commerce / live shopping** surfaces (TikTok Shop, YouTube Live, Shopify Live). It is the reference for proof-safe commerce copy and broadcast structure.
- **Product education / "how it works"** sections for physical products, via `product_education_callout_v1` and the labelled `product_schematic_svg_v1`.
- **Shoppable video** layouts, via `shoppable_video_queue_v1` (CSS/SVG placeholders, never embeds).
- **Honest offer comparison**, via `offer_bundle_matrix_v1` — the go-to when a task needs pricing tiers *without* fabricated discounts.

## Cross-pack donations (use atoms, not the whole page)

- **Any vertical needing an objection FAQ:** donate `objection_faq_panel_v1` (native, keyboard-accessible, placeholder-safe). Re-tone colors to the host system.
- **Any process/agenda/itinerary:** donate `live_demo_sequence_v1` (timecoded timeline) — works for events, onboarding flows, course outlines.
- **Any publishing/QA gate:** donate `platform_tagging_checklist_v1` — adaptable to any pre-launch checklist, not just social platforms.
- **Any explainer block:** donate `product_education_callout_v1` for a numbered mental-model row.

## How to support without conflict

- This is a **cool / dark / tech-commerce** register. When donating into a **warm or light** pack (garden, medspa, beauty, legal), take the *structure and the proof discipline* but drop the ice-blue palette and the broadcast red — remap to the host tokens. Do not import the navy gradient or the glassmorphism wholesale.
- When donating into a **high-contrast industrial/combat** pack, keep only the atom skeletons (timeline, FAQ, checklist); the schematic shimmer and live pulse will read wrong there.
- Preserve every `[… — confirm before publishing]` placeholder and the "illustrative" schematic caption on donation. The atoms are reusable precisely because they carry no claims; stripping the placeholders breaks the proof policy.

## Anti-support (do not pull this pack here)

- B2B SaaS dashboards / admin consoles — wrong surface entirely.
- Static brochure sites with no commerce or demo intent — the broadcast scaffolding is dead weight.
- Anything requiring real social proof or press: this pack deliberately has none, and must never be the donor for a "logos / reviews / stats" request. Route those to a pack with operator-supplied verified evidence instead.

## Operator handoff

When an operator supplies verified evidence (real prices, tested reach/noise figures, real schedule, real clips, real return policy/support contact), swap it directly into the matching placeholder. Until then, the proof-safe copy is the shipped state and should not be "filled in" with estimates.
